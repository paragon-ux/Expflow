# Architecture Decisions

**Status:** Gate D implementation decision register
**Workflow SSOT:** `docs/architecture/EXPFLOW_WORKFLOW_CURRENT.md`
**Source basis:** `EXPFLOW_CONCEPT_PAPER_V2_3.md`, `EXPFLOW_IMPLEMENTATION_SPEC_V2_3.md`, `EXPFLOW_PROTOCOL_SPEC_V2_3.md`, `EXPFLOW_PROJECT_SNAPSHOT_V2_3.md`, `Note-On-Architecture.md`, `V2_3_REVIEW_RESOLUTION.md`, `V2_3_ARCHITECTURE_DELTA.md`

These decisions freeze repository and runtime boundaries as each owning gate becomes implemented.

Gate A freezes invariants and registers future decision slots. It must not force later gates to settle implementation choices before the owning runtime surface exists.

## Decision Maturity Model

Gate A uses four maturity classes:

| Class                 | Meaning                                                                       | Gate A handling                                                                            |
| --------------------- | ----------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------ |
| Frozen invariant      | A rule that later gates must not reopen without an architecture change.       | Record as an architecture decision and verify through registries or tests where practical. |
| Future decision slot  | A concrete implementation choice that belongs to a later gate.                | Name the slot, owner gate, and constraints, but do not decide it prematurely.              |
| Deferred outside core | A contract that belongs to a separately versioned package or adapter profile. | Keep absent from core and verify the boundary.                                             |
| Must not decide yet   | A choice that would imply product runtime behavior before the owning phase.   | Document the boundary and stop expansion pressure.                                         |

## Deferred Decisions

The core repository does not decide adapter request canonicalization, external idempotency grammar, external revision cursors, lost-response reconciliation, adapter capability policy, or writer partitioning. Those decisions belong to separately versioned adapter profiles.

## Executable Decision Vectors

Machine-readable decision vectors live in `registries/decision-vectors.json`. They are contract data for validation and review, not runtime configuration.
